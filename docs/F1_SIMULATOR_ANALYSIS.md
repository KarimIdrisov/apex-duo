# Анализ внешнего проекта: neomat-prog/f1-simulator

Разбор открытого репозитория <https://github.com/neomat-prog/f1-simulator>
(npm-пакет `f1-sim` v1.0.8, MIT, автор `neomat-prog`, последний коммит
2025-04-08). Цель — понять, что в нём есть, как устроена симуляция, и что из
этого полезно (или показательно) для Apex Duo.

- **Дата:** 2026-06-10
- **Статус:** аналитика, код Apex Duo не меняется.
- **Связанные документы:** `docs/COMPETITOR_ANALYSIS.md` (игры-аналоги),
  `docs/F1_DATA_SOURCES.md` (источники данных), `docs/SIM_ARCHITECTURE.md`
  (наша симуляция).

## TL;DR

Это **маленькая JS-библиотека** (~400 строк исходников без `node_modules`),
а не игра: Node.js-пакет, который симулирует гонку «по кругам» (lap-by-lap)
и умеет подтягивать реальный состав пилотов через **OpenF1 API**. Главный
вывод: **README обещает в разы больше, чем реализовано** — safety car,
аварии, DNF, DRS, динамическая погода и свойства шин либо заглушки, либо
мёртвый код. Для Apex Duo проект не конкурент и не источник механик: наша
тиковая модель (0.25 с, ERS/SoC, борьба колесо-в-колесо, детерминизм)
на порядки глубже. Единственная действительно интересная идея — **интеграция
OpenF1 API** как источника реальных данных (состав, квалификация) и приём
нормализации квали-времён в рейтинг машины 80..100.

## 1. Что это и как устроено

```
f1-simulator/
├── src/
│   ├── index.js                 # re-export core + api
│   ├── core/classes/
│   │   ├── Race.js              # вся симуляция (~200 строк)
│   │   ├── Track.js / Team.js / Driver.js   # тривиальные data-классы
│   │   └── index.js
│   ├── api/
│   │   ├── f1-client.js         # OpenF1: session key, состав пилотов
│   │   └── cache.js             # LRU-кэш (создан, но нигде не используется)
│   └── utils/
│       ├── helpers.js           # TireManager, SafetyCarSystem, DRSSystem
│       └── config.js            # точная копия helpers.js (дубль)
├── test/simulator.test.js       # 2 jest-теста
├── examples/basic-usage.js
└── node_modules/                # 68 МБ закоммичены в репозиторий
```

Модель данных минимальна: `Team {name, carPerformance}` (одно число 80..100),
`Driver {name, team, consistency, driverNumber}`, `Track {name, minLapTime,
maxLapTime, drsZones}`.

### Модель гонки (Race.js)

Симуляция **по кругам**, не по тикам: на каждом круге каждому пилоту
начисляется время круга, позиции — сортировка по `totalTime`.

Время круга:

```
carLapTime = maxLap - (perf - 80)/20 * (maxLap - minLap)   # perf 80..100 → лин. интерполяция
tirePenalty = 0.5 + 0.1 * lapsOnTires                       # хардкод, состав шин игнорируется
lap = (carLapTime + tirePenalty) * weatherMult (+20 при SC) # weatherMult: dry=1.0, иначе 1.1
    + U(-1,1) * 0.5 * (1 - consistency/100)                 # шум от «стабильности» пилота
```

Пит-стоп: по заранее заданному словарю `{lap: compound}` — меняет лейбл шин и
обнуляет `lapsOnTires`. **Без потери времени на пит-лейн** — то есть в этой
модели питы строго выгодны, стратегия вырождается в «пить каждые N кругов».

### Интеграция OpenF1 (f1-client.js)

- `getSessionKey(circuit, year, session)` → `/v1/sessions`.
- `getDrivers(sessionKey)` → `/v1/drivers`, собирает `Driver`/`Team` с
  захардкоженной таблицей силы команд сезона ~2023 (Red Bull 95 … Haas 82).
- `Race.initialize()` тянет `/v1/qualifying` и маппит квали-времена в
  `performance = 80 + 20 * (maxLap - lapTime)/(maxLap - minLap)` — стартовая
  решётка по позициям квалификации.

## 2. Заявлено vs реализовано

| Фича из README | Реальность |
| --- | --- |
| ✅ Real F1 data integration | Частично: состав/сессии работают; но эндпоинта `/v1/qualifying` **в OpenF1 нет** (есть `sessions`, `laps`, `position`…) — режим «реальной квалификации» падает/возвращает пусто |
| 🛞 Tire degradation modeling | Линейный `0.5 + 0.1·laps`, одинаковый для всех составов. `TireManager` с soft/medium/hard существует в utils, но **не подключён** |
| 🚨 Safety car probabilities | `checkEvents()` — **пустая заглушка**; `isSafetyCarOut` всегда false. `SafetyCarSystem` в utils не импортируется. Конфиг из README (`SafetyCarSystem.probabilities = {...}`) ни на что не влияет — функция читает локальную константу |
| 🌀 DRS simulation | `DRSSystem` существует, **нигде не используется** |
| 💥 Crash / mechanical failures | `retired` никогда не становится `true`; DNF из примера в README невозможен |
| 🌦️ Dynamic weather + `weatherChanges` | Параметр принимается и **игнорируется**; погода — статический множитель 1.1 |
| 🧠 Pit strategy planning | Работает, но без штрафа времени за пит (см. выше) |

## 3. Замеченные дефекты (помимо нереализованных фич)

1. **Дублирование классов:** `Race.js` заново объявляет и экспортирует свои
   `Track/Team/Driver` поверх отдельных файлов классов; при `export *` из двух
   мест — конфликт имён. Инлайновый `Track` не знает про `drsZones`.
2. **`config.js` = копия `helpers.js`** (полный дубль кода).
3. **README/пример не соответствуют API:** результаты содержат `totalTime`,
   а пример и README печатают `result.time.toFixed(3)` → `TypeError`.
4. **`fetchQualifyingData` перезаписывает `track.minLapTime/maxLapTime`**
   квали-временами — границы трассы «плывут» от данных сессии.
5. **Кэш мёртвый:** `lru-cache` создан, но ни один запрос его не использует.
6. **`node_modules` (68 МБ, ~8800 файлов) закоммичены** в git.
7. **Тесты слабые:** второй тест (`retired.length >= 0`) тавтологичен —
   проходит всегда; «crash-prone driver» ни на что не влияет, т.к. аварий нет.
8. **Нет сидируемого RNG** — `Math.random()` повсюду: прогон невоспроизводим
   (см. контраст с нашим инвариантом детерминизма ниже).

## 4. Сравнение с Apex Duo

| Аспект | f1-simulator | Apex Duo (`race_sim.gd`) |
| --- | --- | --- |
| Шаг симуляции | 1 круг | тик 0.25 с, 3 фазы (темп → борьба → круг/питы) |
| Позиции | сортировка по сумме времени; обгон «бесплатный» | hold-up модель: догнал → строишь pass credit против `_pass_resist()` трассы |
| Энергия 2026 | нет | SoC батареи, clipping, Overtake-boost, active aero |
| Шины | один линейный износ | составы + износ + cliff |
| Надёжность/DNF | нет (заглушка) | reliability-взвешенный ролл, ×1.6 в push |
| Safety car | заглушка | `sc_prob` трассы, bunching, дешёвый пит, отдельный events-RNG |
| Детерминизм | нет (`Math.random`) | сидированный LCG, 2 потока RNG — основа неткода и harness |
| Данные | OpenF1 API (live) | статические таблицы `f1_2026.gd` (engine+chassis) |
| Масштаб | библиотека, ~400 строк | игра: сезон, R&D, кооп-неткод, ~8000 строк |

По всем игровым механикам наша симуляция строго глубже — заимствовать из
кода нечего.

## 5. Что полезно взять на заметку

1. **OpenF1 API** (<https://openf1.org>) — бесплатный источник реальных данных
   (сессии, пилоты, круги, позиции, погода). Для Apex Duo это потенциальный
   инструмент **калибровки** наших таблиц (`REAL_TRACKS`, `CHASSIS`/`ENGINES`):
   реальные распределения времён кругов, деграда шин, частота SC по трассам.
   Добавить в `docs/F1_DATA_SOURCES.md`, если ещё не учтён. В рантайм игры API
   тянуть не нужно — ломает детерминизм и оффлайн-режим.
2. **Нормализация квали-времён в рейтинг 80..100** — простой и наглядный
   приём пересчёта «время → сила машины»; похожая линейная нормализация может
   пригодиться при калибровке `CAR_K`/тиров команд по реальным данным.
3. **Антипример — разрыв README и кода.** Половина фич в README не существует.
   Для нас это подтверждение рабочего правила: README прототипа обновляется
   только по факту реализации, а каждая механика подтверждается численно
   (harness / headless-прогоны) до записи в документацию.

## 6. Вердикт

`f1-simulator` — учебный pet-проект ранней стадии: честная per-lap модель
времени круга с шумом и реальный состав из OpenF1, но почти все «фичи»
из README — заглушки, код содержит дубли и расхождения с документацией.
Как референс игровых механик для Apex Duo не подходит; ценность — только
указатель на OpenF1 API как источник данных для калибровки баланса.
