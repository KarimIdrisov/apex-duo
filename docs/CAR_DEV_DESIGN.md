## 0. TL;DR — что мы строим

Разработка болида превращается из «купи уровень детали за RP» в **глубокий
симулятор проектов** с тремя слоями (ДВС → шасси/аэро → трансферные детали),
двухосевым компромиссом на каждом проекте (темп ↔ надёжность, и шкала риска),
двумя раздельными ресурсами (часы аэротестов ATR — истекают; деньги под cost cap —
видимы и не блокируют), и **встроенным анти-снежным комом** из 5 слоёв. Каждое
решение принимается двумя со-директорами в паддоке: слоты проектов ограничены,
есть превью эффекта, голосование и право вето.

Три цели по приоритету (заданы заказчиком):
1. **Анти-снежный ком / баланс** — лидер не убегает, ИИ не отстаёт, оптимальная
   стратегия совпадает с интересной.
2. **Ко-оп решения вдвоём** — разработка это переговоры двух людей.
3. **Экономика и риск** — износ/надёжность/DNF, замены, cost cap без софт-локов.

Реализм FIA 2026 — фоновая «приправа», а не приоритет: используем его термины
(гомологация ДВС, ATR-шкала, классы компонентов LTC/Transferable, active aero,
50/50 ICE-электро), но не жертвуем балансом ради симуляции.

## 1. Принципы (наследуем из META_DESIGN)

1. **Каждое решение — внятный компромисс между двумя числами.** Игрок видит
   ставку до нажатия кнопки. Нет «абстрактного прогресса».
2. **У каждой системы есть кооп-момент.** Системы спроектированы так, чтобы два
   со-директора естественно расходились и были вынуждены договариваться.
3. **Симуляция остаётся изолированной.** Весь R&D влияет на гонку ТОЛЬКО через
   5 существующих скаляров болида (`d_aero / d_power / d_energy / d_ch_rel /
   d_eng_rel` через `team_car()` в `f1_2026.gd`), скаляры персонала
   (`personnel.gd`) и `morale_mod()`. Ни одной новой записи в `race_sim.gd`.
4. **Числа живут в `const`-таблицах наверху файла** (конвенция проекта). Балансные
   правки — только там.
5. **Детерминизм — несущая конструкция.** Любой «бросок» (тупик проекта, разброс
   результата) идёт через сид-RNG (отдельный поток, как `erng`), чтобы Python-
   харнесс и netcode оставались воспроизводимыми.

## 2. Диагноз: чему учат другие игры

Сводка исследования — что воровать и чего избегать.

**Воровать (лучшие идеи):**

| Идея | Откуда | Зачем нам |
|---|---|---|
| Improvability как отдельный **потолок**, не стартовое значение | Motorsport Manager | «Можно развить дальше, а не начать выше» — длинное решение без мгновенной инфляции темпа |
| Цикл **Дизайн → Производство → Установка** | F1 Manager | Реализм + растягивает решение во времени, создаёт экономику производства |
| **Превью эффекта/веса/ресурса** до коммита | F1 Manager | Идеально под ко-оп: оба директора видят ставку и спорят |
| **ATR**: часы аэротестов истекают, шкала по месту в КЧ | F1 Manager + реальный регламент | Структурный анти-снежный ком (хуже команда → больше часов) |
| Слайдер **темп ↔ надёжность** + кривая отказа | MM + GPRO | «Довести до ~80% надёжности, потом темп» — интуитивный ритм, связь с DNF |
| Ресет регламента **с сохранением порядка** (±разброс), не «−10% всем» | Motorsport Manager | Освежает пелотон, но не обнуляет ощущение прогресса за сезон |
| **Копирование разработки соперника** через роль штаба | iGP (Tech Director) | Анти-снежный ком + ложится на `personnel.gd` |
| **Дилеммы пред-сезона** (2–3 карты компромиссов) | Motorsport Manager | Дешёвый контент, ко-оп-голосование |
| Износ детали + **4-выбор** (улучшить/заменить/откатить-на-менее-изношенную/ничего) | GPRO | Связывает детали с гоночной стратегией и штрафами |

**Избегать (на чём игроки горят):**

1. **Непрозрачный cost cap, который софт-локает.** Худший провал F1M: упёрся в
   кап с деньгами на счету — нельзя построить запасную деталь, нельзя снять
   машину, сезон убит. Игрокам нужно было 4–5 рестартов, чтобы понять систему.
   → У нас cap **мягкий, всегда видимый бегущий итог, предупреждение до коммита,
   никогда не блокирует выход на гонку.**
2. **RNG наказывает игрока (аварии vs cap).** Стоимость запасных деталей зачётом
   в кап → невезучий сезон необратим. → У нас аварии/износ **не уводят экономику
   в спираль смерти**; всегда есть дешёвый «беги на изношенном» путь.
3. **ИИ не умеет развиваться → снежный ком → тривиальный эндшпиль.** Самая частая
   долгосрочная жалоба на F1M. → ИИ развивается по **той же ATR-кривой**, плюс
   рубербанд.
4. **Иллюзия глубины / доминирующая стратегия.** В MM ~47.5% перформанса в
   мотор/КПП → «остальные детали бессмысленны». → У нас **каждая развиваемая
   деталь обязана влиять на реальной трассе** (аудит «нет игнорируемых деталей»).
5. **Игрока заставляют самоограничиваться ради интереса.** Когда оптимальная игра
   тривиализует → игроки сами себя «гимпят» или ставят моды. → **Весёлый путь и
   оптимальный путь совпадают.**
6. **Ресет стирает прожитый прогресс** («−10% всем» = наказание). → См. п. о
   порядок-сохраняющем ресете.
7. **Потерянная валюта без обратной связи** (истёкшие часы WT/CFD, сгоревшие DP).
   → Истечение ATR **явно подсвечивается**, излишку дан сток (см. «банк часов»).

## 3. Архитектура: трёхслойная модель болида

Болид разбит на три слоя по тому, **как** они развиваются. Это отражает классы
компонентов FIA 2026 и, главное, даёт три разных типа решений.

```
┌────────────────────────────────────────────────────────────────┐
│ СЛОЙ A — СИЛОВАЯ УСТАНОВКА (ДВС/PU)         → d_power, d_energy,  │
│   решение РАЗ В СЕЗОН: выбор поставщика +                d_eng_rel│
│   гомологация (заморожена в сезоне) + пул компонентов + штрафы    │
├────────────────────────────────────────────────────────────────┤
│ СЛОЙ B — ШАССИ / АЭРО (LTC, только своя)    → d_aero, d_ch_rel    │
│   ОСНОВНОЙ ЦИКЛ: проекты разработки каждый этап,                  │
│   часы ATR, слайдеры компромисса, производство, установка        │
├────────────────────────────────────────────────────────────────┤
│ СЛОЙ C — ТРАНСФЕРНЫЕ ДЕТАЛИ                 → d_ch_rel (+ малый    │
│   ось КУПИТЬ-vs-РАЗВИТЬ: КПП, гидравлика,            d_aero)      │
│   подвеска. Покупное = надёжно/дёшево/потолок ниже;               │
│   своё = дорого/долго/потолок выше                               │
└────────────────────────────────────────────────────────────────┘
```

Почему три слоя, а не один список деталей: каждый слой даёт **разный ритм
решения**. PU — редкое стратегическое (раз в сезон, тяжёлые последствия). Аэро —
частое тактическое (каждый этап, основная петля). Трансферные — позиционное
(купить сейчас потолок-пониже или вложиться в своё).

## 4. Модель детали (глубокий сим)

Каждая **развиваемая** деталь (слой B и C) имеет ПЯТЬ свойств вместо одного
`level`. Это сердце «глубины».

| Свойство | Диапазон | Что делает | Канал в сим |
|---|---|---|---|
| `perf` (уровень темпа) | 0.0 … `ceiling` | Текущий вклад в свой скаляр | d_aero / d_ch_rel / … |
| `ceiling` (потолок / improvability) | напр. 0.6 … 1.0 | Максимум, до которого можно развить в этой эре регламента | ограничивает рост |
| `expertise` (экспертиза проекта) | 0.0 … 1.0 | Накоплённое знание → дешевле/быстрее будущие проекты этой детали | множитель стоимости |
| `reliability` (надёжность детали) | 0.0 … 1.0 | Отдельно от темпа; низкая → DNF | d_ch_rel / d_eng_rel |
| `condition` (износ) | 0.0 … 1.0 | Падает каждый этап; чинится заменой | масштаб вклада (уже есть) |

Ключевой принцип (из MM): **`perf` и `ceiling` независимы.** Хороший базовый
болид (высокий стартовый `perf`) ≠ болид с большим потенциалом (`ceiling`).
Аутсайдер может стартовать ниже, но иметь больший потолок и ATR-фору — и догнать.

**Что поднимает `ceiling` (решение D):**
- Аэро-детали (слой B) — **инфраструктура**: прокачка Аэротрубы / CFD / Дизайн-
  центра поднимает аэро-потолок (и параллельно открывает слоты проектов, §11).
  Деньги в здания = долгосрочная инвестиция в потолок против сиюминутного
  производства деталей. Классический читаемый рычаг.
- PU-детали (слой A) — **поставщик**: `improvability` выбранного PU-поставщика
  задаёт потолок мотора на сезон (§7).
- Ресет регламента (§9) поднимает потолки всем (новая эра).

**Экспертиза — на деталь, не на ветку (решение E):** `expertise` копится
**по каждой детали отдельно**. Качал днище → дешевле качать днище дальше, но не
переднее крыло. Создаёт выбор «добить знакомую деталь дёшево или открыть новую
дорого» и поощряет осмысленную специализацию.

`condition` уже реализован (`compose_part_deltas` принимает словарь condition,
floor 60%, критическая <0.30 бьёт надёжность). **Сохраняем как есть.**

### Расширенный список деталей (16 шт.)

Берём текущие 12 из `f1_2026.gd PARTS` и достраиваем до 16, сохраняя структуру
`group / scalar / per_level / max_level / also / also_rel / buy_cost`. Аудит «нет
игнорируемых деталей»: каждая привязана к характеристике трассы, чтобы на разных
этапах ценились разные детали (как в F1M — Монако ≠ Монца).

**Аэро (LTC, слой B) — только своя разработка:**

| Деталь | Ключ | Скаляр | Профиль трассы (где ценна) |
|---|---|---|---|
| Переднее крыло | `front_wing` | d_aero | медленные повороты (downforce) |
| Заднее крыло | `rear_wing` | d_aero | баланс прижим/срыв (drag) |
| Днище | `floor` | d_aero | быстрые повороты, граунд-эффект |
| Боковые понтоны | `sidepods` | d_aero + охлаждение | жаркие/мощностные трассы |
| Геометрия подвески | `suspension_geo` | d_aero | поребрики, тряска (Сингапур) |
| Монокок (новое) | `monocoque` | d_aero + d_ch_rel | базовая жёсткость, влияет на всё |

**ДВС/энергия (слой A) — через поставщика, гомологация:**

| Деталь | Ключ | Скаляр | Примечание |
|---|---|---|---|
| ДВС | `ice` | d_power | 2026: ~536 л.с. ICE |
| Турбо | `turbo` | d_power | |
| Батарея | `battery` | d_energy | 2026: 50% мощности электро |
| MGU-K / ERS | `ers` | d_energy | 2026: 350 кВт деплой |

**Трансферные (слой C) — купить-vs-развить:**

| Деталь | Ключ | Скаляр | buy_cost |
|---|---|---|---|
| КПП | `gearbox` | d_ch_rel (+малый d_aero) | есть (120k) |
| Гидравлика | `hydraulics` | d_ch_rel | есть (100k) |
| Охлаждение | `cooling` | d_ch_rel (+малый d_power) | — |
| Тормоза (поставщик) | `brakes` | d_ch_rel + pit_cons | сезонный поставщик (есть) |
| Топливо/масло (поставщик) | `fuel` | d_power + d_energy | сезонный поставщик (есть) |
| Дифференциал (новое) | `differential` | d_ch_rel | трансферный |

> **Балансный коридор (НЕ нарушать).** Суммарный размах внутрисезонного аэро-
> развития ≤ **0.12–0.15** d_aero (как сейчас: 6 шагов × 0.025). Полный размах PU-
> развития ≤ **0.03–0.05** d_power/d_energy (сейчас полный gap PU ≈ 0.0125, коридор
> 0.010–0.015). Дельты надёжности ±**0.03–0.05**. Это держит `team_car()` в
> текущем масштабе 0..1 (~0.78 базовая аэро/мощность) — иначе ломается CAR_K и
> DNF-кривая.

## 5. Проект разработки — ядро петли

Игрок не «покупает уровень». Он **запускает проект** на деталь. У проекта три
органа управления — все читаемые, все с превью.

### Орган 1: Слайдер «Темп ↔ Надёжность» (alloc ∈ [0,1])

Распределяет вложение между приростом `perf` и `reliability` детали.
- `alloc = 1.0` → весь прирост в темп, надёжность не растёт (риск DNF).
- `alloc = 0.0` → весь в надёжность, но конструктор всё равно даёт малый трикл
  темпа (из MM — даже all-reliability проект чуть двигает perf).
- Типичная стратегия (намеренно поощряем): сначала довести до ~80% надёжности,
  потом гнать темп. Это ритм, а не доминирующая стратегия — потому что на разных
  трассах и при разном везении с DNF оптимум плавает.

```
perf_gain  = base_gain × alloc × diminish × expertise_mult × risk_mult
rel_gain   = base_rel  × (1 − alloc) + REL_TRICKLE
```

### Орган 2: Шкала риска (target ∈ {safe, normal, aggressive})

«Насколько агрессивную концепцию пробуем».
- `safe`: ×0.8 прироста, разброс ноль, тупик невозможен.
- `normal`: ×1.0, малый разброс.
- `aggressive`: ×1.4 прироста, НО детерминированный бросок:
  - с шансом `P_DEADEND` (растёт у топ-команд — см. анти-снежный ком) проект
    **тупиковый**: даёт ~0.3× ожидаемого и сжигает часы/деньги;
  - delivered `reliability` ниже на DEADEND_REL_MALUS.

Бросок через отдельный сид-RNG (поток `drng`, сид из `mix32(seed ^ part ^ round)`),
детерминизм сохранён.

### Орган 3: Глубина проекта (часы/время)

Сколько часов ATR и какой срок (этапов) вложить. Дольше = больше `perf_gain` и
больше накопленной `expertise` (из F1M: длинные проекты дают больше экспертизы,
эффективнее держать один проект, чем дробить).

### Кривая затухания (diminish) — анти-снежный ком #2

```
diminish = clamp( (ceiling − perf) / (ceiling − perf_soft_knee), 0, 1 )
```
Чем ближе `perf` к `ceiling`, тем дороже каждый следующий пункт. У потолка прирост
→ 0. Лидер, выкачавший деталь, упирается в стену; аутсайдер на «дешёвом» участке
кривой растёт быстро.

### Превью эффекта — кооп-момент #1

Перед коммитом панель показывает (как в F1M, но для двоих):
```
Проект: Днище, ур. 1 → 2   |  Срок: 2 этапа  |  Часы ATR: 38/100
Ожидаемо:  d_aero +0.018   надёжность +0.004   вес −0.3
Риск: aggressive → шанс тупика 18%, при удаче +0.025
Стоимость производства: $420k (в cap)  |  Остаток cap: $3.1M ✅
[ Директор P5: ⬜ за ]   [ Директор P6: ⬜ за ]   нужно согласие обоих
```
Оба видят ставку и **спорят до траты общего ресурса**.

## 6. Экономика: два раздельных ресурса

Намеренно держим **два** ресурса — это и есть «экономика и риск» как приоритет.

**Философия дефицита — асимметричная (решение A, уточнено харнессом).** Главный
вывод проверки кошелька (`wallet.py`): **под ATR объём разработки ограничен часами
у всех**, поэтому деньги **не конвертируются в скорость напрямую**. Реальные стены
разные по тирам:
- **Топ-команда** упирается в **часы ATR** (лидер получает лишь 0.70× базы =
  меньше всех часов, несмотря на самый большой бюджет) и в **капекс-лимит** на
  здания. Её денежный профицит **инертен** — купить ещё скорости нельзя (cost cap
  + капекс-лимит не дают). Это и есть защита от денежного снежного кома.
- **Аутсайдер** богат **часами** (до 1.15×) — это его догоняющий рычаг — но
  **тесен по деньгам на операционке** (производство/апкип/разведка съедают бо́льшую
  часть дохода). Решает, на что хватит, но развивается быстрее за счёт часов.

То есть экономика — реальная ось (выбор у аутсайдера, тайминг зданий, батчинг
производства, разведка, зарплаты), но **не она стена топа**: топа держат часы и
капекс. Конкретные числа — в §6.1 (утверждены харнессом).

### 6.1. Числа единого кошелька (утверждено харнессом `wallet.py`)

Сезон 25 этапов / 6 окон. Доход — существующие покруговые коридоры `season.gd`
(прайз по месту + спонсоры), × 25 этапов. Зарплаты пилотов/топ-штаба — **вне
cost cap** (как в реальной F1).

| Тир | Старт-бюджет | Доход/сезон | Зарплаты/сезон | Часы ATR/окно | Стена |
|---|---|---|---|---|---|
| Контендер | **6.0M** (было 8M) | ~21.9M | ~10.25M | 70 | часы ATR + капекс |
| Середняк | **5.0M** | ~15.5M | ~7.0M | 90 | смешанная |
| Андердог | **4.0M** (было 3M) | ~11.25M | ~3.75M | 115 | часы ATR (деньги тесны) |

**Капекс-лимит на здания** (отдельный от операционного cost cap — реальная F1:
capital expenditure имеет свой потолок) — слегка скользит по месту, усиливая
догоняние: контендер 2.5M, середняк 3.0M, андердог 3.5M за сезон. Это ограничивает
~1–2 апгрейда зданий в сезон любому, и не даёт богатому мгновенно отстроить базу
(§7.5, лаг постройки).

**Замечание (см. аудит §17):** профицит контендера (~10M инертных денег) —
осознанная цена реализма дохода по месту. Деньги сверх капекс+операционки идти в
скорость не могут; излишек уходит на звёздных пилотов (вне капа) и переносится в
следующий сезон. Альтернатива при желании — сжать прайз-кривую; параметр в §14.

### 6.0. Удаление RP — единая денежная валюта

**Решение: RP удаляется из игры полностью** (нереалистична — в настоящей F1 нет
абстрактных «очков»). После переработки в игре остаётся:

- **Деньги** (под cost cap) — единственная валюта на всё: производство деталей,
  замены, поставка PU, поставщики, **зарплаты персонала (M2)**, **академия/юниоры
  (M5)**, разведка/реверс-инжиниринг (§10).
- **Часы ATR** — не валюта, а возобновляемый «бюджет аэротестов» под разработку
  болида (слой B + конвергенция PU).

Что это затрагивает (объём миграции, для фазы P1):
- Хаб (`season_hub.gd`) и все экраны, где показан RP, переводятся на деньги.
- Развитие персонала/академии (`personnel.gd`, M2/M5) переходит на денежную
  оплату; их балансы перетюниваются под единый кошелёк.
- Штраф cost cap раньше списывал RP → теперь урезает **часы ATR** следующего окна
  (см. Ресурс 2 ниже).
- Загрузчик сейвов: старое поле `rp` игнорируется/конвертируется (разовая миграция,
  как было с `part_levels`).

Риск миграции: деньги становятся «узким горлом» всего — нужен аккуратный
ребаланс доходов (призовые КЧ + спонсоры M1), чтобы и железо, и люди влезали в
один кошелёк. Это явный пункт тестирования фазы P1.

### Ресурс 1: Часы аэротестов (ATR) — истекают, шкала по месту

Это **основной R&D-ресурс** аэро-разработки (слой B). Реальная основа — ATR-шкала
FIA 2026. (Старая валюта **RP удалена** — см. §6.0.)

- **Окно развития** = блок из ~4 этапов календаря. При сезоне в **25 этапов** это
  даёт **6 аэро-окон** за сезон (как реальные 6 двухмесячных ATR-периодов 2026).
  Решений по разработке ~6 за сезон — весомых, не утомляющих; проекты тянутся 1–2
  окна. (Параметр `ATR_WINDOW_RACES = 4`; для коротких exhibition-календарей окно
  масштабируется автоматически.)
- Каждое окно начисляет `ATR_BASE` часов (напр. **100**), умноженных на
  **скользящую шкалу по месту в Кубке конструкторов прошлого периода**:

  | Место в КЧ | 1 | 2 | 3 | … | 10 | 11 |
  |---|---|---|---|---|---|---|
  | Множитель ATR | 0.70 | 0.75 | 0.80 | +0.05/место | 1.15 | 1.20 |

  Это **главный структурный анти-снежный ком**: лидер получает 70% часов,
  аутсайдер — 120%.
- **Истекают**: неиспользованные часы периода частично сгорают. НО есть
  **банк-сток** (из урока «не теряй валюту молча»): до `ATR_BANK_CAP` (напр. 20%)
  переносится на следующее окно; всё сверх — сгорает с явным предупреждением
  («⚠ 34 ч ATR сгорят в конце окна — запустите проект или перенесите 20 ч»).

### Ресурс 2: Деньги под Cost Cap — видимый, мягкий, без локов

- Деньги платят за **производство** деталей, **замены** изношенных, поставку PU,
  поставщиков. ATR — это «можно ли спроектировать», деньги — «можно ли построить».
- **Бегущий итог cap всегда на экране.** Перед каждым коммитом — превью остатка.
- **Мягкое правило (наследуем META-3):** превышение cap не блокирует, а даёт
  штраф — урезание **часов ATR следующего окна** (`CAP_PENALTY_DIVISOR`; раньше
  пеня шла в RP — теперь, без RP, бьём по темпу разработки). **Никогда** не мешает
  выйти на гонку и **никогда** не запрещает построить запасную деталь.
- **Дешёвый аварийный путь всегда есть:** «беги на изношенной детали» (риск DNF/
  потеря темпа) бесплатен. Невезение не уводит в спираль смерти.

### Цикл Дизайн → Производство → Установка (из F1M)

1. **Дизайн** (тратит ATR, занимает срок): проект «созревает», даёт `perf_gain` и
   `expertise`.
2. **Производство** (тратит деньги/cap): спроектированную деталь надо построить —
   отдельный платёж. Можно отложить (копить несколько апгрейдов и поставить
   пакетом — из F1M: ждать и ставить вместе выгоднее, чем по капле).
3. **Установка** (на уик-энде): фитим деталь на болид. До установки — старая.

## 7. Слой A: ДВС / Силовая установка (PU)

PU — **сезонное стратегическое решение**, не часть еженедельной петли. Это
реализм FIA 2026 (гомологация: спецификация ДВС заморожена на сезон) и одновременно
анти-снежный приём (нельзя бесконечно качать мотор в сезоне).

### Выбор поставщика (раз в сезон)

В межсезонье выбираешь PU-поставщика (как сейчас движки общие: Ferrari питает
Ferrari/Haas/Cadillac). Поставщик — это **набор модификаторов** (из MM):
- `base_power`, `base_energy`, `base_rel` (стартовые d_power/d_energy/d_eng_rel),
- `improvability` (потолок развития PU-деталей в сезоне),
- `cost` (поставка за сезон, реальная ≈ €17M),
- `partner_pay` (доход от тех-партнёрства — закрывает спонсорский слот M1).

Компромисс: топ-поставщик = темп сейчас, но дорого и низкий потолок роста;
бюджетный = слабее старт, выше improvability + свободные деньги под аэро-cap.

### Гомологация и развитие PU в сезоне

- PU-детали (`ice/turbo/battery/ers`) развиваются **ограниченно** и **дорого**
  внутри сезона (узкий коридор ≤ 0.03–0.05), отражая заморозку. Бо́льшая часть
  прыжка по мотору происходит **между сезонами** (новая спецификация).
- Это удерживает PU-гонку от снежного кома: нельзя выехать на одном моторе.

### Конвергенция ДВС (реальный механизм 2026) — анти-снежный слой для PU

Реальный регламент 2026 предусматривает **механизм догоняния по мощности**:
производителю ДВС, который существенно отстал по мощности, FIA разрешает
**дополнительные апгрейды/часы стенда внутри сезона**, чтобы сойтись с лидерами;
лидеры по PU при этом **заморожены** до межсезонья. Моделируем напрямую:

- Каждое окно считается **PU-дефицит** твоего мотора относительно лучшего
  PU-поставщика в пелотоне (по композитному d_power+d_energy).
- Если дефицит выше порога `PU_CONVERGE_GAP` (напр. 0.025), команде/производителю
  открывается **право на PU-проект внутри сезона** — можно развивать мотор по
  обычной петле (ATR/деньги) в пределах коридора конвергенции.
- Чем больше отставание, тем выше разрешённый шаг (но не выше уровня лидера —
  **догнать можно, перегнать заморозкой нельзя**).
- Лидер по PU своё развитие в сезоне **не ведёт** (заморожен) — у него остаётся
  только аэро-петля.

Эффект: автоматический рубербанд именно по мотору. Слабый PU не приговор на весь
сезон, но и не превращается в способ обгона — только сходимость. ИИ-производители
конвергируют по тем же правилам. (Связь: дополняет анти-снежный ком §8.)

### Пул компонентов и штрафы (из реальной F1 + GPRO)

- На сезон даётся **пул** PU-компонентов (напр. 3 ДВС, 3 батареи и т.д.).
- Каждый новый компонент сверх пула = **штраф на старте** (потеря позиций на
  гриде следующей гонки). Изношенный компонент можно «дотянуть» (риск надёжности)
  или заменить (тратит пул).
- **Кооп-момент #2:** пул PU **общий на обе машины**. Если у P5 отказывает мотор и
  он берёт 4-й — штраф влияет на стратегию всей команды. Два инженера должны
  координировать расход пула (кому новый мотор сейчас, кто терпит).

## 7.5. Инфраструктура — здания, потолки и слоты

Раз потолок аэро и число слотов проектов завязаны на здания (решения D и слоты
§11), инфраструктура — отдельная система долгосрочных инвестиций. Это **второй
полюс экономики**: каждый рубль в здание — это рубль не в производство деталей
сейчас. Фантазия: «мы строим команду, которая будет быстрой через сезон».

### Принципы

1. **Капекс ≠ гонка.** Здания — капитальные вложения, отделённые от еженедельной
   возни с деталями (реальная F1: capital expenditure имеет отдельный лимит вне
   cost cap). В игре это один денежный кошелёк (RP удалён), но здания —
   **крупные разовые лампы**, конкурирующие с производством и зарплатами.
2. **Лаг постройки (анти-снежный ком).** Апгрейд здания строится **1–2 окна** и
   даёт эффект только после. Лидер не может мгновенно конвертировать деньги в
   доминирование — инвестиция окупается позже, что сглаживает разрыв.
3. **Затухающая отдача.** Каждый следующий уровень здания дороже и даёт меньше
   (как diminish у деталей). Нельзя «выкупить» 5-й уровень в отрыв.
4. **Потолок ≠ часы.** Аэротруба поднимает *качество* (потолок + отдача на час),
   но **количество часов всё равно режет ATR-шкала по месту** (§6). Лидер с лучшей
   трубой получает мало часов → отдача его трубы ограничена. Это ключевая связка,
   которая не даёт зданиям сломать баланс.

   > **Проверено численно (2026-06-17).** Лидер (КЧ-1, ATR 0.70) с **максимальным**
   > CFD-центром даёт 0.0109 d_aero/окно; аутсайдер (КЧ-10, ATR 1.15) даже со
   > **слабым** CFD — 0.0146/окно (быстрее на +0.0037). Если у обоих CFD на максимуме,
   > аутсайдер быстрее на +0.0070. Вывод: ATR-шкала (0.70↔1.15) перевешивает весь
   > диапазон множителя зданий (1.0↔1.30) — инфраструктура не даёт снежного кома.

### Набор зданий (5 шт.)

Все эффекты — **выше по течению** от 5 скаляров (меняют потолок/часы/слоты/цены,
не сим напрямую). Опираемся на 4 здания F1M (Factory / Design Centre / Wind Tunnel
/ CFD) + разведотдел под §10.

| Здание | Ось | Эффект за уровень | Канал |
|---|---|---|---|
| **Аэротруба** (Wind Tunnel) | потолок | +`AERO_CEILING_PER_FACILITY` к аэро-потолку | `ceiling` слоя B |
| **CFD-центр** | эффективность | ×множитель к `GAIN_PER_HOUR` (больше d_aero на час) | отдача ATR |
| **Дизайн-центр** (Design Centre) | слоты + знание | +1 слот проекта (до `PROJECT_SLOTS_MAX`); +скорость накопления `expertise` | §11 слоты, §4 |
| **Завод** (Factory) | производство | дешевле/быстрее производство деталей; +деталей в окно | §6 экономика |
| **Разведотдел** | информация | дешевле/точнее разведка; на высоком ур. +1 цель «прожектора» | §10 |

Профили выбора: **Аэротруба** — длинная ставка на потолок (окупается к концу
сезона); **CFD** — немедленная отдача на каждый час (хорош аутсайдеру с большим
ATR); **Дизайн-центр** — больше параллельных проектов и дешевле повторная
разработка; **Завод** — режет производственные расходы (важно топу под cost cap);
**Разведотдел** — включает догоняющую игру через копирование (§10).

### Механика апгрейда

- Уровни **0..5** на здание. Стоимость растёт (`FACILITY_COST[level]`), отдача
  затухает.
- Постройка занимает `FACILITY_BUILD_WINDOWS` (1–2 окна); пока строится — старый
  уровень.
- Малый **апкип** за окно (`FACILITY_UPKEEP`) держит денежное давление: большая
  база требует содержания (ещё один рычаг против бесконтрольного разрастания).

### Стартовые здания по тиру команды

Завязываем на выбор тира в `season_setup` (контендер/середняк/андердог):
- **Контендер** — топ-здания (труба/завод 4–5), но мало ATR-часов и тугой
  cost cap → апгрейды дорогие, упор на эффективность.
- **Андердог** — слабые здания (1–2), но ATR-фора и свободные деньги → именно ему
  выгодно вкладываться в трубу/CFD и расти.
Это делает стартовый разрыв «отыгрываемым»: у топа лучше база, у андердога —
лучше траектория роста.

### Кооп-момент

Во что вложить ограниченные деньги — спор двух со-директоров: **труба** (потолок
через сезон) против **завода** (дешёвые детали сейчас) против **дизайн-центра**
(ещё один слот, чтобы вести оба их проекта разом). Решение по зданию — общий
коммит (консенсус, §11).

### Маппинг в сим

Здания не трогают `race_sim.gd`. Они меняют параметры R&D-петли:
`ceiling` (Аэротруба), `GAIN_PER_HOUR` (CFD), число слотов и `expertise`-rate
(Дизайн-центр), цену/скорость производства (Завод), параметры разведки
(Разведотдел). Всё это уже сворачивается в 5 скаляров через §13.

## 8. Анти-снежный ком — 6 слоёв (приоритет #1)

Самый частый провал жанра — лидер убегает, ИИ глупый, эндшпиль тривиален. Ставим
**шесть независимых тормозов**, чтобы ни один не нёс всю нагрузку.

1. **ATR-шкала по месту (§6).** Хуже команда → больше часов разработки (0.70 для
   1-го → 1.20 для 11-го). Структурный, всегда активный.
2. **Кривая затухания у потолка (§5).** Лидер у `ceiling` платит всё дороже за
   пункт; аутсайдер на дешёвом участке. Прирост у потолка → 0.
3. **Шанс тупика у топов на `aggressive` (§5).** `P_DEADEND` растёт с местом в КЧ:
   лидер чаще ловит тупиковую концепцию, гоняясь за последними десятыми. Риск-
   премия снижается по мере доминирования.
4. **Ресет регламента с сохранением порядка (§9).** Периодически сжимает пелотон,
   не обнуляя прогресс.
5. **Разведка + копирование (§10).** Отстающая команда (и ИИ) может через скаута
   вскрыть и реверс-инжинирить недавний апгрейд лидера со скидкой. Догоняющий
   механизм по аэро.
6. **Конвергенция ДВС (§7).** Отстающий по мотору получает право на PU-развитие в
   сезоне; лидер по PU заморожен. Догоняющий механизм по силовой установке.

**ИИ развивается по тем же правилам.** Используем существующий хук
`apply_ai_dev()` (`f1_2026.gd`) и ATR-кривую: каждый ИИ-соперник копит дельты на
своей ATR-доле (его место → его множитель). ИИ **не тупой**: он распределяет часы
по приоритетным деталям трассы-фаворита и поднимает надёжность до порога. Игрок
обыгрывает ИИ умом (тайминг пакетов, выбор трасс), а не тем, что ИИ не умеет
развиваться.

### Числовая проверка анти-снежного кома (целевой коридор)

Цель — на «Обычной» сложности разрыв лидер↔аутсайдер по суммарному скаляру болида
**сходится, а не расходится** за сезон. Ориентир: к середине сезона топ-3 в
пределах **±0.03** d_aero друг от друга; аутсайдер сокращает отставание на
**0.01–0.02** d_aero за период (за счёт ATR-форы), но не обгоняет без удачных
проектов. Проверяется Python-харнессом (см. §13).

> **Проверено численно (2026-06-17).** Прогон зеркала §14: старт-разрыв 0.080
> d_aero (лидер 0.090 @ КЧ-1, аутсайдер 0.010 @ КЧ-10). За 8 периодов аутсайдер
> прибавляет стабильно ~0.012–0.015/период, прирост лидера затухает с 0.009 до
> 0.003 у потолка, разрыв сходится к **0.013–0.031** — без обгона по чистому
> темпу. Подтверждает: ATR-шкала + затухание дают сходимость, а не расхождение.
> Ключевой балансный множитель — `GAIN_PER_HOUR ≈ 0.00012` (при 0.0006 обе
> команды упираются в потолок за 2 периода — слишком быстро).

## 9. Ресет регламента — с сохранением порядка (из MM)

Когда суммарное развитие пересекает порог эры (или по календарю — большая смена
регламента), **следующее межсезонье** ресетит `perf` деталей к базовой линии
**с разбросом ±SPREAD**, грубо сохраняя пелотон-порядок; детали **ниже минимума не
ресетятся** (из MM: 2850/1350/600 для WMC — у нас в скаляр-космосе свои значения).

Почему не «−10% всем» (F1M): плоский вайп ощущается как наказание и стирает
прожитый сезон. Порядок-сохраняющий ресет **освежает** (новые потолки, новые
дилеммы), но игрок чувствует, что заработанное место осталось при нём.

`ceiling` при ресете растёт (новая эра — выше потолок), `expertise` частично
переносится (накопленное знание команды не испаряется), `condition` сбрасывается
в 1.0.

## 10. Туман войны, разведка и копирование (из iGP + MM)

Соперники **не прозрачны**. Это делает разработку информационной игрой, а не
оптимизацией с открытыми картами.

### Туман войны

По умолчанию о сопернике видно только **примерный рейтинг машины** (1–5 «звёзд»
по композиту скаляров, с шумом) и место в КЧ. Какие именно детали и на каком
уровне он развил — **скрыто**. Свой болид виден полностью.

### Скаут (новая мини-система)

Роль штаба — **Скаут / разведчик** (ложится на `personnel.gd`). Действие
**«Разведать команду»** за деньги вскрывает на 1 окно реальную карту апгрейдов
выбранного соперника:
- что он поставил, на каком уровне, его сильные/слабые детали по трассам;
- точность раскрытия растёт с навыком скаута (слабый скаут даёт диапазоны, не
  точные числа);
- **«прожектор» = 1 активная цель за окно** (решение B): нельзя вскрыть всех
  сразу, выбираешь одну — обычно лидер (ради копии) или прямой конкурент в КЧ
  (ради тактики). Цена ≈ стоимость одного дешёвого проекта производства.

### Реверс-инжиниринг (гейтится разведкой)

Только **по вскрытому скаутом** апгрейду доступно действие **«Реверс-инжиниринг»**:
за деньги/часы скопировать недавний апгрейд команды **выше по КЧ** со скидкой
относительно собственной разработки.
- Скидка **30–50%** против своей разработки (решение B), тем больше, чем выше навык
  штаба и чем дальше отрыв — помогает именно отстающим.
- Копия приходит с потолком на 0.5 «уровня» ниже оригинала (как покупная
  трансферная деталь, `SUPPLIER_LEVEL_EQ`) — догнать можно, перегнать копией
  нельзя.
- **ИИ тоже разведывает и копирует** игрока-лидера — рубербанд работает в обе
  стороны.

**Кооп-момент:** на что тратить ограниченный «прожектор» разведки — спор двух
директоров (вскрыть лидера ради реверса или прямого конкурента ради тактики?).

## 11. Кооп: разработка как переговоры двоих

Разработка — мета-решение, принимаемое **вместе в паддоке** (не в гонке, где
каждый рулит своей машиной). Механики спроектированы под расхождение мнений.

1. **Доска проектов с ограниченными слотами.** Дизайн-центр даёт N слотов проектов
   одновременно (уровень здания, §7.5). Слотов меньше, чем хочется → выбор
   обязателен.
2. **Чемпионство ветки.** Каждый со-директор «ведёт» свою ветку: напр. инженер P5
   двигает аэро, инженер P6 — надёжность/PU. Но **пул ATR и денег общий** — это
   создаёт естественный спор «твой проект днища или мой проект надёжности».
3. **Голосование и вето — чистый консенсус.** Коммит проекта требует согласия
   обоих (превью §5). **Тай-брейка нет**: если не согласны — проект не стартует,
   ресурс ждёт. Намеренно: со-директора всегда могут договориться вживую, а
   принудительный оверрайд ломал бы дух симметричного ко-опа.
4. **Дилеммы пред-сезона (из MM).** 2–3 одноразовые карты компромиссов («+1 нагрев
   шин / −1 износ», «купить улучшение поставщика за большую цену»). Оба голосуют.
5. **Координация пула PU (§7)** и **замен деталей на уик-энде** — общие на две
   машины, требуют согласования «кому свежий компонент сейчас».

Это прямое продолжение симметричного ко-опа Apex Duo: мета-решения берутся вместе,
в гонке каждый ведёт свой болид.

## 12. Турновый цикл (как это играется)

**Межсезонье (раз в сезон):**
1. Выбор PU-поставщика (§7) — компромисс темп/потолок/деньги.
2. Ресет регламента, если сработал (§9): новые потолки, перенос экспертизы.
3. 2–3 дилеммы (§11), голосование вдвоём.

**Паддок (между этапами — основная петля):**
1. Свериться с балансом ATR: сколько часов, сколько сгорит, что перенести (§6).
2. Доска проектов (§11): выбрать, какие детали развивать в слоты.
3. На каждый проект: слайдер темп/надёжность, шкала риска, глубина (§5).
4. Превью эффекта → голосование/коммит вдвоём.
5. Решить, что **производить** сейчас, что копить пакетом (§6).
6. Буй-vs-девелоп по трансферным (§ слой C); разведка цели + реверс лидера (§10).
7. Координация пула PU и замен (§7).
8. **Инвестиции в здания** (§7.5): начать апгрейд (строится 1–2 окна) — общий
   коммит. Спор «труба на потолок vs завод на дешёвые детали vs слот».

**Уик-энд этапа:**
1. Детали изнашиваются (`condition` ↓, уже есть). Каждый инженер решает по своей
   машине: фитить свежую (деньги/cap) или бежать на изношенной (риск).
2. Расход пула PU → штрафы на гриде при превышении.
3. Гонка (симуляция читает только 5 скаляров).

**После гонки:**
1. Начисляется `expertise` завершённым проектам.
2. Обновляется `condition`, проверка ресета.
3. ATR-период катится: предупреждение об истечении часов, банк-перенос.
4. Доход (призовые КЧ + спонсоры M1) пополняет деньги под cap.

## 13. Маппинг в симуляцию (изоляция)

Всё богатство сворачивается в **5 существующих скаляров**. Никаких новых каналов
в `race_sim.gd`.

```
части (perf, reliability, condition)  ─┐
поставщик PU (base_*, improvability)  ─┼─► compose_*_deltas()  ─► {d_aero, d_power,
трансферные/поставщики (brakes/fuel)  ─┘                            d_energy, d_ch_rel,
                                                                    d_eng_rel}
                                                                          │
                                              apply_rd_upgrades(team) ◄───┘
                                                                          │
                                                          team_car(team_idx) ──► sim
                                          (d_*_rel → DNF-бросок в race_sim, уже есть)
```

- `perf` детали → её `scalar` (как сейчас `per_level × level`, но `level`
  становится непрерывным `perf` с потолком).
- `reliability` детали → `d_ch_rel` / `d_eng_rel` → существующая DNF-кривая
  (`DNF_BASE`, ×1.6 при пуше). **Низкая надёжность буквально = больше DNF.**
- `condition` → существующий масштаб вклада (floor 60%, критическая бьёт rel).
- PU-поставщик → `base_power/energy/rel` складываются в те же скаляры через
  `team_car()` (движок+шасси композятся, как сейчас).
- ИИ-развитие → `apply_ai_dev()` (уже есть), кормится ATR-кривой.

Расширяем `compose_part_deltas()` под непрерывный `perf` и `ceiling`; добавляем
`compose_pu_supplier()` по образцу `compose_supplier_deltas()`. **Сигнатуры выхода
не меняются** — `apply_rd_upgrades()`/`team_car()` трогать не нужно.

## 15. Поэтапное внедрение (фазы)

Внедряем итеративно, каждая фаза — играбельный инкремент с тестом-зеркалом
(Python harness → потом GDScript). Порядок по принципу «сначала самый ценный
анти-снежный слой».

**Фаза P1 — удаление RP + ATR-экономика + затухание (анти-снежный ком #1, #2).**
Убрать RP отовсюду (§6.0), перевести всё на деньги, ввести часы ATR со шкалой по
месту и кривую затухания у потолка. Самый дешёвый и самый сильный балансный рычаг.
Тесты: `cardev_atr_check.py` (сходимость разрыва лидер↔аутсайдер за сезон) +
проверка ребаланса единого кошелька (железо+люди влезают в деньги).

**Фаза P2 — модель детали (perf/ceiling/reliability) + слайдеры проекта +
инфраструктура (§7.5).** Перевести `part_levels` (дискретные) в непрерывный `perf`
с `ceiling`; добавить слайдер темп/надёжность и шкалу риска с детерминированным
тупиком. Расширить `compose_part_deltas()`. Ввести здания (Аэротруба/CFD/Дизайн-
центр/Завод/Разведотдел): потолок, отдача на час, слоты, цена производства; лаг
постройки + апкип; стартовые уровни по тиру. Тесты: `cardev_project_check.py`
(отдача vs надёжность vs DNF) + `cardev_facility_check.py` (лаг и затухание зданий
не дают лидеру убежать).

**Фаза P3 — слой A (PU-поставщик + гомологация + конвергенция + пул/штрафы) и слой C
(буй-vs-девелоп).** Сезонный выбор PU, конвергенция ДВС, пул компонентов,
координация на 2 машины. Опереться на существующие supplier-таблицы (brakes/fuel)
и `compose_supplier_deltas()`. Тест: `cardev_pu_check.py` (включая сходимость по
конвергенции).

**Фаза P4 — ресет регламента + шпионаж + кооп-UI.** Порядок-сохраняющий ресет,
реверс-инжиниринг через `personnel.gd`, доска проектов с голосованием и превью
эффекта. Тест: `cardev_reset_check.py` (порядок сохраняется), плюс реал-движок
прогон через godot MCP.

Каждая фаза: research → числа в Python → имплемент → линт (трюк со свежим файлом) →
обновить README/доки → хендофф (рабочее соглашение саб-агентов).

## 16. Принятые решения (согласовано 2026-06-17)

Все стартовые открытые вопросы закрыты заказчиком:

1. **Календарь = 25 этапов.** Окно развития = ~4 этапа → **6 аэро-окон/сезон**
   (§6, `ATR_WINDOW_RACES = 4`).
2. **Слоты проектов завязаны на инфраструктуру:** старт 2, до 4 через прокачку
   Дизайн-центра/Завода (§11, §14).
3. **Тай-брейка в ко-опе нет** — чистый консенсус, со-директора всегда
   договариваются вживую (§11.3).
4. **PU развивается в сезоне через конвергенцию ДВС** (реальный механизм 2026):
   отстающий по мотору получает право на сезон-развитие, лидер заморожен (§7).
5. **RP удалён полностью** — единая денежная валюта на всё; разработка болида на
   ATR + деньги (§6.0).
6. **Туман войны + разведка:** соперники видны как рейтинг; скаут вскрывает карту
   апгрейдов, что гейтит реверс-инжиниринг (§10).
7. **Активное аэро (Z/X) в car-dev не моделируется** — механика 2026 ещё не в
   симе; разработку на неё не завязываем.

### Уточнения (согласовано 2026-06-17, второй проход)

- **A. Дефицит асимметричный (уточнено харнессом):** топ упирается в **часы ATR +
  капекс** (деньги-избыток инертен), аутсайдер богат часами, но тесен по деньгам на
  операционке (§6, §6.1). Сам по себе анти-снежный ком.
- **B. Разведка:** 1 цель за окно, цена ≈ дешёвый проект; реверс-скидка 30–50% с
  ростом по отрыву/навыку (§10, §14).
- **C. 16 деталей утверждены**, включая `monocoque` (фундамент, редкий дорогой
  проект) и `differential` (дешёвый трансферный) (§4).
- **D. Аэро-потолок поднимает инфраструктура** (Аэротруба/CFD/Дизайн-центр);
  PU-потолок — поставщик (§4, §14).
- **E. Экспертиза копится по детали**, не по ветке (§4).
- **F. Пред-сезонные дилеммы — в фазу P4** (полиш), не в ядро (§15).

### Остаточные числа для харнесса P1 (не блокируют дизайн)

- Точный старт-бюджет/доход под асимметричный профиль дефицита (A).
- Кривая `AERO_CEILING_PER_FACILITY` и стоимость зданий (D).
- Точные `buy_cost` для `differential` и нового `monocoque` (последний — LTC, без
  `buy_cost`).

## 17. Аудит логики (сквозная проверка, 2026-06-17)

Прошёл всю систему на согласованность, балансные коридоры, анти-снежный ком,
противоречия, маппинг в существующий код и краевые случаи. Ниже — что проверено,
найдено и исправлено.

### Численно проверено (Python-харнесс)

1. **Сходимость аэро-разрыва** (`cardev_check`): старт-разрыв 0.080 d_aero сходится
   к 0.013–0.031 за 8 окон; аутсайдер +0.012–0.015/окно, лидер затухает к потолку.
   Анти-снежный ком #1+#2 работает. Ключ: `GAIN_PER_HOUR ≈ 0.00012`.
2. **Здания не дают снежного кома** (`cardev_facility`): лидер с максимальным CFD
   (0.0109/окно) развивается медленнее аутсайдера со слабым CFD (0.0146/окно).
   ATR-шкала (0.70↔1.15) перевешивает диапазон зданий (1.0↔1.30). «Потолок ≠ часы».
3. **Единый кошелёк** (`wallet`): подтвердил асимметрию стен и утвердил числа §6.1.

### Найдено и исправлено в ходе аудита

- **(major) Старый `SALARY_CAP = 4M` был рассчитан на 5 этапов.** При 25 этапах
  зарплаты ~10M/сезон — старый кап бессмыслен. → Заменён на `COST_CAP_SEASON = 6M`
  (операционный) + зарплаты вынесены **вне капа**; пеня капа теперь режет часы ATR
  (§6.1, §14).
- **(major) Деньги топа не должны конвертироваться в скорость.** Харнесс показал:
  контендер копит ~10M профицита. Под ATR разработка ограничена часами у всех →
  деньги в скорость не идут. Добавлен **капекс-лимит на здания** (отдельный от
  операционного капа, как capex-allowance реальной F1), чтобы и через постройку
  богатый не отстроил доминацию мгновенно (§6.1, §7.5).
- **(fix) Уточнена «Философия дефицита» (§6).** Раньше писал «топ упирается в
  деньги/cost cap» — неточно: топа держат **часы ATR + капекс**, а cost cap — в
  основном реализм (производство и так лимитировано часами). Текст исправлен.

### Проверено — согласовано, без правок

- **Изоляция симуляции.** Всё (детали, PU, поставщики, здания, разведка) сворачивается
  в 5 скаляров через `compose_*` → `team_car()`; туман войны/разведка — чисто

- **Балансные коридоры.** Аэро-развитие за сезон: аутсайдер ~0.088, контендер ~0.065

  текущего масштаба `team_car()` (CAR_K/DNF не ломаются). ✓

### Остаточные риски / на тюнинг в P1–P2 (не блокируют дизайн)

- **Инертный профицит контендера (~10M).** Не ломает баланс (деньги ≠ скорость),
  но «пустые» деньги — лёгкий запах. Опции: звёздные пилоты вне капа, перенос на
  капекс след. сезона, или сжать прайз-кривую `CONSTRUCTOR_PRIZE`. Решается тюнингом.
- **Сумма потолков 16 деталей** должна в сумме давать ≤0.15 d_aero — при переводе
  `part_levels`→непрерывный `perf` свести per-part `ceiling` в общий коридор.
- **Ребаланс единого кошелька** после удаления RP затрагивает M2/M5 (персонал,
  академия) — их балансы перевести на деньги и проверить в `cardev_atr_check.py`.
