  **C (UI) и D (netcode) делят `main.gd`** → их правки секвенируем, один агент на

- **DoD фичи:** спека выполнена · числа доказаны в Python · код парсится+линтуется
  (fresh-file) · **прошёл `validate_scripts`** · README/GDD обновлены · ревью ·

## Реконсилиация: что РЕАЛЬНО осталось (аудит против кода, 2026-06-09)

Источник правды — сам код (`race_sim.gd`, `season.gd`, `main.gd`, `f1_2026.gd`), не
roadmap-доки. Несколько «будущих» пунктов уже реализованы (как SIM-2). Ниже — статус,
файл-изоляция, чем верифицировать (движок vs Python-харнесс) и порядок.

### 🔎 SIM-4 — Погода: УЖЕ РЕАЛИЗОВАНА → нужна только engine-верификация

- **Что в коде (`race_sim.gd`):** `WET_K=16.0`, `WET_BASE=3.0`, `wet_opt` у компаундов
  (slick 0 / inter 0.45 / wet 0.85), переменные `wetness`/`wet_start`/`wet_end`/`wet_peak`,
  триангулярный ливень от `track.wet_prob` (детерминизм через events-логику), `_m_wet()`
  штраф за несоответствие шины влажности + смягчение атрибутом `wet` пилота, **ИИ-кроссовер**
  слик↔интер↔вет в стратегии (стр. ~794–809), кнопки дождевой резины игроку (README/TODO).
- **Чего НЕ хватает:** только подтверждения чисел **на реальном движке** (харнесс упрощает
  энергопетлю; погода завязана на dirty-air/SoC). Acceptance из старой спеки SIM-4 (слик
  теряет ≥ 3 c/круг за кроссовером; «дождевой мастер» различим; вовремя сменивший отыгрывает)
  — прогнать через godot MCP, **не переписывать**.
- **Рекомендация:** свернуть в **SIM-VER** (тест-кампания на движке). Кода писать не нужно.
- **Трек/файл:** A · `race_sim.gd` (без правок, если числа сойдутся) · **godot MCP**.

### 🟡 SIM-3 — Обгон по трассам: модель per-track ЕСТЬ → остаётся калибровка

- **Что в коде:** `_pass_resist()` = `4.0 + 9.0·(1 − track.overtaking)` — порог уже
  per-track; `_resolve_combat()` строит `pass_credit` из чистого pace-edge + Overtake-boost
  (масштаб от `track.power`·`track.overtaking`), dirty-air через `DA_COEF` и `track.downforce`;
  каждая реальная трасса несёт свой `ot` (Монако 0.05 … Монца 0.82). Баг «бесконечных обгонов»
  закрыт (README: Монако ~2, Монца ~95 за гонку).
- **Чего НЕ хватает:** это **калибровка**, не новая фича — подвинуть константы под реальные
  коридоры обгонов **на движке** (CLAUDE.md: харнесс это судить не может). Разблокировано MCP.
- **Acceptance (через godot MCP):** Монако ~1–3, Монца ~70–100, остальные между; доля
  «заработанных» обгонов ≈ 100%; Overtake-boost в пределах 1 c измеримо помогает; детерминизм
  по сиду. (Старая спека SIM-3 ниже — она по сути верна, меняется только слово «настройка».)
- **Трек/файл:** A · `race_sim.gd` (константы обгона) · **godot MCP**. **Делит `race_sim.gd`**
  с SIM-VER/META-2b — секвенировать на этом файле.

### ⬜ META-2b — Прошить `attr_dev` в сим (ABSENT, мелкий, высокая отдача)

- **Долг (точно):** `season.gd` считает и сохраняет `attr_dev_of(id, attr)` (per-attribute
  рост за сезон), но `main.gd:_make_sim` (стр. ~162) подаёт в сим **только скаляр**
  `dev_of()` → плоский бонус к `d.skill`. Per-attr рост (вырос `tyre` → должен снижать износ;
  вырос `overtaking` → быстрее `pass_credit`) **не доходит** до `d.attrs[...]`, которые читает
  `_attr()`. Т.е. headline META-2 не замкнут end-to-end. README сам помечает это TODO
  («прошивка attr_dev в сим»).
- **Goal:** в `_make_sim` после `make_field()` для своих пилотов прибавлять `attr_dev_of(id,k)`
  к соответствующему `d.attrs[k]` (в skill-единицах → 1..20-шкала), вместо/в дополнение к
  плоскому `dev_of()` на `d.skill` (избегать двойного счёта — `driver_dev` ↔ `attr_dev` дублируют).
- **Acceptance (Python-харнесс + движок):** пилот с выросшим `tyre` показывает **различимо
  меньший износ** в симе; с выросшим `overtaking` — **больше обгонов**; суммарный эффект ≈
  старому скалярному `dev_of()` (без инфляции силы); детерминизм по сиду; регресс зелёный.
- **Трек/файл:** B · правка в **`main.gd` (~162)** (это `_make_sim`, не HUD — точечно) +

- **Effort:** **S**. **Зависит от:** META-2 ✅.

## SIM-1 — Квалификация + грид/разброс на старте

- **Goal:** добавить квалификационную сессию, формирующую грид, и разброс на
  первом круге, вместо чисто skill-сортированной расстановки.
- **Acceptance criteria (числа в Python-харнессе):**
  - Квала: порядок грида коррелирует со скиллом+характером машины на трассе, но с
    шумом — **ранг-корреляция поула с чистым скиллом ≈ 0.75–0.92** (не 1.0:
    возможны сюрпризы), детерминизм по сиду сохранён (тот же сид → тот же грид).
  - Старт: разброс P1 круга даёт реальные перестановки — **в среднем ≥ 2 позиции
    меняют владельца на круге 1** по сидам; разброс зависит от атрибута `starts`
    пилота (хорошие старты выигрывают позиции чаще, метрика различима между
    архетипами).
  - Грид-эффект на гонку: поул конвертируется в победу не автоматически —
    **win-rate поул-ситтера в диапазоне 35–60%** по сидам на трассе среднего
    обгона (не 100%, не 0%).
  - Регресс: существующие проверки харнесса остаются «зелёными».
- **Track / роли:** A. researcher (формат квалы 2026 — кратко), designer (feel:
  насколько «случаен» старт), programmer, tester.
- **Файлы:** `race_sim.gd` (новая квала-фаза + старт-разброс; `GRID_GAP` уже есть),

## Рекомендуемый порядок к 3D-финалу (после реконсилиации)

Волны 1–2 закрыты. Цель — **доводка существующего на движке → enabler-рефактор →
3D-финал**, максимально параллеля файл-изолированные треки.

### Волна 3 — «ЗАПУСКАЕМ СЕЙЧАС» (2 параллельных потока в worktree)

1. **SIM-VER** (трек A, `race_sim.gd` + godot MCP) — **главный приоритет**. Поглощает
   **engine-верификацию SIM-4** (погода уже в коде) и **калибровку SIM-3** (обгон уже
   per-track) в один engine-поток, и закрывает блокер «полный движковый прогон
   `race_sim.gd` не делался». Сюда же удобно подцепить **CAL-1** (калибровка по данным F1).
   *Один поток на `race_sim.gd` — не параллелить с другими sim-правками.*
2. **META-2b** (трек B, правка в `main.gd:_make_sim` ~162 + harness) — **S**, замыкает
   META-2 end-to-end (per-attr рост реально влияет на сим). Касается `main.gd`, но
   точечно и не в HUD; если REF-1 ещё не сделан — это безопасная локальная правка.

**Почему так:** SIM-VER даёт **уверенность в числах живого движка** (а не харнесса) —
это разблокирует всё дальнейшее и снимает риск, что волны 1–4 где-то не сходятся.
META-2b — дешёвый «замок» на уже сделанную META-2, высокая отдача за S.

### Волна 4 — enabler + контент (параллельно)

3. **REF-1** (трек C/D) — расколоть `main.gd` → `net.gd` + `hud.gd`. **Делать до 3D-3 и
   до тяжёлого UI-1/коопа** — иначе интеграция 3D и NF-* идёт в монолит. Отдельный
   worktree, обязательный ревью + ручной play-check.
4. **META-3** (трек B/C, после META-1 ✅) — cost cap + контракты; B-логика в `season.gd`,
   C-UI в `season_hub.gd` (**не `main.gd`** → безопасно). Файл-изолировано от A.
5. **3D-1 / 3D-2** (трек C, **новые файлы** `track_shapes.gd`/`track_builder_3d.gd`/
   `race_view_3d.gd`) — можно начинать **параллельно** REF-1/META-3: они в новых файлах,
   контеншена нет до фазы интеграции. 3D-1 численно верифицируема в харнессе.

### Волна 5 — 3D-финал + кооп-глубина (после REF-1)

6. **3D-3 → 3D-7** (трек C) — интеграция в `hud.gd`, болиды/состояния, камеры/атмосфера,
   split-screen, полировка. 3D-3 и 3D-6 — единственные точки контеншена (`main.gd`/`hud.gd`).
7. **UI-1** + **NF-1/NF-2** (телеметрия-асимметрия, пинги — USP-кооп) — после REF-1,
   секвенировать на `hud.gd`/`net.gd` с 3D-3/3D-6.
8. **NF-3/NF-4/NF-5** (риск-ось, VSC, эволюция трассы) — трек A, после SIM-VER, тот же
   `race_sim.gd` (секвенс).

|-------|------------------------|----------------------------|
| **3 (сейчас)** | SIM-VER (A, `race_sim.gd`+MCP) · META-2b (B/`main.gd` точечно) | SIM-VER единолично владеет `race_sim.gd` |
| **4** | REF-1 (C/D) · META-3 (B/C `season.gd`+`season_hub.gd`) · 3D-1/3D-2 (C, new files) | REF-1 раскалывает `main.gd` — **до** 3D-3 |

**Жёсткие правила секвенса (неизменны):**
- **`race_sim.gd`** (SIM-VER/SIM-3/SIM-4-verify, NF-3/4/5, CAL-1) — **один поток** или
  раздельные worktree с аккуратным мёрджем (один файл, детерминизм — несущая стена).
- **`season.gd`** (META-3, WKND-1) — то же.
- **`main.gd` / `hud.gd` / `net.gd`** (META-2b, REF-1, 3D-3/3D-6, UI-1, NF-1/2, META-4) —
  **один агент за цикл мёрджа**. **REF-1 раскалывает монолит и снимает контеншн** —
  поэтому он перед 3D-интеграцией и кооп-UI.
- **A ↔ B параллельны всегда** (файл-изолированы). 3D-геометрия (новые файлы) параллельна
  всему до фазы интеграции.

### Где нужна engine-истина (godot MCP) vs Python-харнесс

- **godot MCP (движок) обязателен:** SIM-VER, SIM-3 (калибровка обгона), SIM-4 (верификация
  погоды), CAL-1, NF-4 (VSC-частота), **любой визуал 3D** (3D-3…3D-7 — только в редакторе
  у человека), финальный баланс инцидентов/контактов. *Причина (CLAUDE.md): харнесс упрощает
  SoC/энергопетлю и dirty-air↔Overtake — обгон/погоду/контакты он судить не может.*
- **Python-харнесс достаточно (verify-first):** META-2b (per-attr → износ/кредит как
  изолированная метрика), META-3 (симуляция бюджета на 5 этапов), NF-3/NF-5 (износ/темп),
  3D-1 (геометрия `Curve3D`: длина/шаг/замыкание). Затем точечная engine-проверка.
