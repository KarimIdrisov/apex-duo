## Приоритет: player-experience

### 1. Лента комментариев гонки (event-feed)
**Откуда:** Race-Engine (`commentary.py`) — лучший takeaway по вовлечённости.
Структурированный фид радио/комментатора: события из сима → лента сообщений на race-screen.
- Источник событий уже есть в `sim.js`: старт, перелом `_passCredit` (→ «обгон!»),
  вход в питы (`_serveLapEnd`), сход (`retired`), быстрый круг (sessionBest mini),
  лидер после старта.
- Реализация: отдельный `src/commentary.js` (чистые функции `genX() -> string`),
  сим/`order()` пушит коды событий, `ui/race.js` рендерит ленту.
- Проверяемо юнит-тестом (событие X при условии Y).
- **Замечание:** это «новая система» — согласовать с фокусом «чинить/тюнить» перед стартом.

### 2. Забег до 1-го поворота (перетасовка позиций на старте)
**Откуда:** Race-Engine `calculate_run_to_turn1`.
`time_to_turn1 = dist_to_turn1 / (avg_speed + random)`, +N м на каждое место грида →
честный сдвиг позиций на старте, а не только бинарный «инцидент/нет».
Сейчас `gridStart()` просто разносит по скиллу; добавить стартовый шаффл в первый круг.

### 2a. Co-op мини-игры на пит/заправку/настройку (идея, нужен дизайн)
**Откуда:** idougherty/f1 (`minigames/` — fuelmixer и пр.). Единственная *новая* идея из
присланных ссылок, бьющая в вовлечённость 2 игроков, а не в калибровку.
Превратить рутинные действия (питстоп, смесь топлива, попадание в окно сетапа) в быстрые
интерактивные мини-задачи, которые игрок делает вживую. Для симметричной коопы — у каждого
свой мини-челлендж на своей машине; результат → бонус/штраф к pit-времени или setup-близости.
- **Каркас уже есть:** practice-пазл сетапа (`ui/practice.js`) — та же идея «мини-задача → бонус».
- **Большой скоуп / новая система** — нужен дизайн-проход; конфликтует с фокусом «чинить/тюнить».
- Реализацию idougherty НЕ брать (шуточные ассеты), только концепт.

### 2b. Зоны обгона (где/когда случается пас) — ПЛАН
**Зачем:** дать обгону *текстуру* («обгон на торможении в 1-й поворот!») без позиционной
физики/траекторий. Сейчас бой разрешается «где-то на круге» по pass-credit + `track.ot`;
зоны привязывают завершение паса к реальным местам обгона. Ложится на готовые мини-секторы
(`track.js`: `MINI`/`N_MINI`/`miniSplits`). НЕ моделирует боковое положение/линии — это
сознательно (игрок-инженер этим не управляет; детерминизм/netcode не трогаем).

**Модель данных (в `data.js`, per track):**
- `overtake_zones: [{ sectors:[i..j], ease:0..1, type:"brake"|"slip" }]` — 1–3 зоны на трассу.
  `ease` = локальная обгоняемость (заменяет глобальный `track.ot` внутри зоны; вне зоны пас
  не завершается). `brake` = точка торможения, `slip` = конец прямой (слипстрим).
- Барселона стартово: 2 зоны вручную (T1 после старт-прямой; вход в финальный сектор).

**Изменение боя (`sim.js _resolveCombat`):**
- pass-credit копится как сейчас (из едж по темпу) ВЕЗДЕ в пределах `COMBAT_GAP`.
- Завершение паса теперь требует: преследователь в зоне (его `lapFrac`→мини-сектор∈zone.sectors)
  И credit > `resist=(1−zone.ease)·k`. Вне зоны — держим прижатым (как сейчас), credit копится
  («получил выхлоп на прямую»). Слипстрим `SLIP_K` → к `slip`-зонам, грязный воздух → к поворотным.
- ИНВАРИАНТ сохраняется: бой пишет ТОЛЬКО `lapFrac`, `lap` не трогает (фикс из памяти).

**Хук комментария:** при завершении паса известна зона → событие `{type:"overtake", zone, a, b}`
→ кормит ленту комментариев (п.1). Прямая синергия.

**Источник зон (две опции):**
- (a) руками 1–3 зоны на трассу (просто, но 25 трасс);
- (b) ПОЛУ-АВТО из кривизны: `tools/fastf1_extract.py` уже считает curvature по трассе →
  длинные низко-кривизные сегменты = прямые = кандидаты в `slip`-зоны, точки перед резким
  ростом кривизны = `brake`-зоны. Синергия с калибровкой; снимает ручную разметку 25 трасс.

**Фазы:** A) данные + Барселона руками → B) бой выбирает зону → C) хук комментария →
D) (опц.) авто-зоны из кривизны для остального календаря.

**Тесты/риски:** детерминизм (тот же сид → тот же результат) обязателен; число обгонов за гонку
должно остаться в калиброванном коридоре `ot` — **зоны кластеризуют пасы, балансы надо
перетюнить** (меньше «случайных» обгонов в твисти-секциях); новый тест: пасы происходят в
секторах зон. **Новая механика** — перед кодом зайти через brainstorm, не на ходу.

## Приоритет: калибровка (FastF1)

### 3. FastF1-extractor → реальные числа трасс
**Контекст:** веб-сим не привязан к 2026-регламенту → реальная телеметрия 2023–2025
ложится почти 1:1 (оговорка про 2026 здесь НЕ действует).
Python-скрипт (в духе `tools/balance.mjs`-workflow): по трассе выдаёт целевой JSON
`{lt, pw, df, ot, sc, abr, pit, compound_deltas, weightK}` из реальных сезонов.
- Соответствие: `lt`←pole/race time; `pw`/`df`←top-speed vs corner-min из speed-трейса;
  `ot`←реальные обгоны/гонку; `sc`←история SC; `abr`←деградация стинта;
  `pit`←pit-loss; `COMPOUNDS.pace/cliff`←дельты Pirelli.
- **Окупается при расширении на N трасс** — генерить набор трассы из данных, не руками.
- Первый шаг: заземлить Барселону, сверить с нынешними `lt:80, pw:0.55, df:0.82, ot:0.30`.
- fastf1 3.8.3 работает здесь (PYTHONIOENCODING=utf-8).

**СТАТУС (v2 готов, валидирован):** `tools/fastf1_extract.py` — 2 режима:
`--gp Spain` (детально) и `--tracks Monza Monaco Spain ...` (нормировка pw/df в 0..1).
- ✅ `lt` подтверждён: real p20 79.45 / median 80.6 vs наш 80.0 (per-track lt переносим как есть).
- ✅ `pw/df` ПОЧИНЕН (физика кривизны `v²·κ`, клип 6.5g): Барселона → pw−df −0.34 AERO,
  сходится с ручным −0.27. Monza корректно POWER (+1.0). v1-баг (full-throttle → POWER) устранён.
- ✅ `compounds` ПОЧИНЕН (глобальный fit отделил топливо): fuel −0.062с/круг, знаки верны
  (soft −0.22 / hard +0.52); наш hard в точку, soft-гэп −0.55 завышен → можно ужать до ~−0.3.
- ✅ `pit` ПОЧИНЕН: per-driver green-референс убрал прогрев out-lap (Барселона 25.3→23.4с, n=42).
- ✅ `ot` ДОБАВЛЕН: on-track обгоны через lap-by-lap position diff (искл. старт+пит-циклы);
  Барселона 0.67 passes/lap; нормируется в 0..1 в `--tracks`/`--calendar`.
- ⚠️ df-шкала = min-max по набору — стабилизируется на полном календаре (идёт прогон).
  Monaco df завышен туннелем; Monza df=0 — артефакт минимума набора.
- Кэш fastf1 — в системном temp, не в репозитории.
- **Режимы:** `--gp <GP>` (детально), `--tracks A B C` (нормировка), `--calendar` (все 24 GP 2024,
  пишет `tools/track_constants_2024.json`).
**ПОЛНЫЙ КАЛЕНДАРЬ 2024 ПРОГНАН** → `tools/track_constants_2024.json` (24 трассы):
- ✅ КАЛИБРОВОЧНОЕ КАЧЕСТВО, готово к переносу: `lt` (70.4–107.8, все чисты), `pw/df`
  (POWER=Mexico/Monza/Baku, AERO=Monaco/Suzuka/Zandvoort), `ot` (Monaco 0.0 … Vegas 1.0).
- 🟡 `pit`: относительный порядок ок, абсолют завышен для длинных кругов (Britain/Singapore).
- ❌ `compounds`: guard отсёк мусор, но выжило 12/24 и они слишком шумные (soft median +0.32,
  неверный знак). ВЫВОД: per-race стинты не годятся для темпа резины (определяется контекстом
  гонки, не резиной). Брать quali/practice low-fuel или Pirelli; ручные S−0.55/H+0.55 НЕ менять.
**Готово к data.js (когда дашь добро — это правка игрового кода):** per-track `lt/pw/df/ot` из JSON
при расширении за Барселону; `pit` как относительный. Компаунды оставить как есть.

**Компаунды — ОКОНЧАТЕЛЬНО (режим `--compounds`, low-fuel из FP/Q):** чистый темп резины из
FastF1 НЕ извлекается. Race-стинты спутаны топливом (soft +0.32, неверный знак). Low-fuel best-laps
спутаны эволюцией трассы (Q3-софт vs практика-медиум) + топливом (медиум/хард в практике на тяжёлом
баке) → выходит ВДВОЕ завышено (soft −1.3 vs реальных ~−0.7). Компаунды всегда едут на разном
топливе/состоянии трассы — извлечь нельзя. ИСТОЧНИК: только опубликованные дельты Pirelli или тест.
Плюс: soft −1.3 / hard +1.25 симметричны → симметрия нашей модели −0.55/+0.55 подтверждена.
Ручные значения НЕ менять.

**Мульти-сезон (`--years`, #3):** усредняет сырые сигналы (top_speed/aero_load/passes/lt/pit) по
сезонам, потом нормирует один раз → стабильная шкала df/ot. Полный календарь 2022–2024 ПРОГОНЯЕТСЯ
(фон) → `tools/track_constants_multiseason.json`. (Одна ground-effect эра, усреднение валидно;
отсутствующие в сезоне трассы — Vegas'22/China'22-23/Qatar'22 — пропускаются gracefully.)
**Готовые к переносу в data.js (когда скажешь):** per-track `lt`; направление pw/df (Барселона AERO
подтверждён); ужать `COMPOUNDS.soft.pace` −0.55 → ~−0.3. Правок data.js НЕ делал (код игры не трогаю).

**3a. Bradley-Terry → driver-skill из реальных результатов** (из F1-PREDICT `fit_bt_model.py`).
Сейчас `TEAMS[].drivers[].skill` проставлены руками (Норрис 0.950 и т.д.). BT-модель выводит
силу пилотов **эмпирически** из попарных исходов сезона (кто кого обогнал), с поправкой на

**3b. Калибровочные метрики в `balance.mjs`** (из F1-PREDICT `calibration_check.py`/`temp_scale.py`).
Сейчас балансные коридоры проверяются на глаз. Строже: прогнать сим много сидов → win-probability

### 5. Физический lap-sim как офлайн-калибратор (опционально)
**Откуда:** maxtayloraero F1 lap-sim (QSS: кривизна по сегментам → предельные скорости →

---
**Не брать:** Race-Engine как референс симуляции (он слабее нашего: по-круговой шаг,
линейный износ без компаундов, плоские 20% DNF, без детерминизма). PathTrackingBicycle
(Stanley/PID/bicycle-model) и self-driving-f1 (NEAT/GA/ES/DQN, 2D-машина с радарами по

ML-serving, telemetry-replay), не игровой сим; ценны только методы → вынесены в 3a/3b.

без поля/обгонов, износ/топливо как множители времени круга, без детерминизма) — слабее
нашего по всем осям. Единственная смежная тема — погода (dry/rainy/extreme), но она уже
