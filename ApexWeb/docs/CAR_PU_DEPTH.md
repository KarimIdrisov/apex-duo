# Углубление разработки болида и ДВС (E1–E4)

> **Рефактор (2026-06, схема v22).** Система оказалась переусложнённой. Упростили:
> убрали **аэро-ресурс/ATR** (разработка гейтится деньгами + слотами + кост-капом) и
> **двухстадийный пайплайн проект→сборка** (проект снова одностадийный: развивается N дней →
> устанавливается). Вкладку «Машина» переразбили на под-вкладки **Шасси / ДВС / Стратегия**:
> размер и подход выбираешь один раз сверху, у каждой детали — одна кнопка «Развивать»
> (а не 3×6 зелёных кнопок); характеристики — мини-барами. Осталась вся интуитивная глубина:
> подход (риск↔надёжность), исходы, потолок/зрелость, обкатка, концепт, ресурс ДВС, фокус
> этот/след. год, экономика ИИ. Кнопок на вкладке стало 33→14, высота 842→542px.

Статус: реализовано (2026-06). Схема сейва v22. Превращает «заплати → жди → +фикс»
в набор настоящих решений. Всё детерминировано (сиды), кооп-безопасно; влияние на сим
идёт только через индикаторы машины (`applyRaceMods`) — ядро сима не трогается.

## E1 — параллельная разработка, риск, надёжность, поломки в гонке
- **Несколько проектов сразу.** Слотов = `1 + floor(factory/2)` (`maxProjects`). Завод
  больше — больше параллельных программ. Один проект на деталь за раз.
- **Подход к разработке** (`APPROACH`): консервативный / сбалансированный / агрессивный —
  масштабирует целевой прирост, разброс исхода и долг надёжности.
- **Исход** (`projectOutcome`, сид): провал / частично / успех / прорыв. Агрессивный
  расширяет ОБА хвоста (проверено: safe 5/5% по краям, aggressive 20/20%).
- **Затухающая отдача** (`maturityFactor`): чем выше уровень детали к потолку
  (`PART_CEILING`), тем меньше прирост (1.0 → 0.5 → 0.15). Нет бесконечной прокачки.
- **Необкатанные детали → поломки в гонке.** Свежая деталь несёт долг надёжности
  (`unproven`), который **снижает `rel` машины в гонке** → выше шанс схода (сим:
  `DNF_BASE*(1-rel)*…`). Долг гаснет за `RUNIN_RACES` гонок (`runInParts`). Агрессивный
  подход и провал дают больший долг. То есть рискованный апгрейд реально может сломаться.

## E2 — аэро-ресурс (труба / ATR)
Второе ограничение помимо денег. Каждый проект тянет ёмкость по размеру (`AERO_COST`).
Пул **регенерится за гонку обратно месту в чемпионате** (`aeroRegen`: лидер ~11/гонку,
последний ~19) и растёт с КБ — честная скользящая шкала ATR (отстающим больше тестов),
не скрытая «резинка». Проект нельзя начать без ёмкости.

## E3 — концепт болида на сезон
Выбор философии (`CONCEPT`): прижимной / сбалансированный / мощностной. Дельта
**симметрична** (+аэро/−мощность или наоборот) — не меняет абсолютный темп
((power+aero)/2), а сим сам взвешивает её трек-характером
(`(power−aero)×(pw−df)`): прижимной быстр в поворотах (трассы прижима), мощностной —
на прямых. Держится весь сезон; смена бесплатна в межсезонье, $3.5M по ходу сезона.
Синергия с календарём (смотришь, каких трасс больше).

## E4 — ДВС как ресурс сезона
Лимит силовых установок (`PU_POOL = 4`). Каждая гонка изнашивает текущий ДВС
(`puWearForRace`): мощностные трассы — быстрее (pw 0.9 → 0.224/гонку), трассы прижима —
медленнее (pw 0.2 → 0.112); **развитая надёжность ДВС продлевает ресурс**
(rel-дев 0.25 → 0.176). Спалил ресурс — ставится следующий ДВС; вышел за пул → **штраф
`PU_GRID_PEN = 5` мест на старте** следующей гонки (применяется в `main.startRace`,
сдвигает машину игрока по гриду). Нейтральный сезон ≈ ровно 4 ДВС — то есть мощностной
календарь или отсутствие разработки надёжности ведут к штрафам; развивай надёжность,
чтобы их избежать.

## Интеграция и поток
`main.buildField/practiceCars`: машина игрока = `composeCar(applyRaceMods(effectiveCarPU(...),
career, track))` — `applyRaceMods` накладывает долг надёжности (E1) и концепт (E3).
`career.applyResult`: `runInParts` (обкатка) + регенерация аэро (E2) + износ ДВС (E4).
`career.advanceRound`: `tickDevelopment(career, gapDays)` крутит все проекты по дням.
UI — вкладка «Машина»: слоты, аэро-ресурс, подход, зрелость деталей, активные проекты,
предупреждение о необкатанных, концепт, ресурс ДВС; в карточке след. этапа — штраф ДВС.

## Проверено (браузер, реальные файлы)
Исходы по подходам, затухание зрелости, ATR по позиции, `applyRaceMods` (rel 0.92→0.88,
концепт power ±0.05), обкатка 2→1; UI: слоты 1/2→2/2 и аэро 50→20 при старте проекта,
концепт-панель, ресурс ДВС 1/4 · 100%; миграция v17→v19. Износ ДВС сверен по формуле.

## E6 — пуш-режим в гонке изнашивает ДВС
Сим считает долю времени машины игрока в режиме «push» (`pushTicks/runTicks`,
накапливается в тик-цикле `sim.step`). Доля передаётся через `main` → `applyResult`
→ `puWearForRace(track, puRel, pushFrac)`: полный пуш-режим всю гонку добавляет
+0.7 к стрессу (≈+70% износа). Теперь то, КАК жмёшь мотор в гонке, прямо влияет на
ресурс ДВС и риск штрафа — петля «гонка ↔ менеджмент сезона» замкнута.

## E7 — концепты у ИИ-команд
Каждая ИИ-команда несёт стабильный концепт (`aiConcept`): команды ранжируются по
уклону базовой машины power−aero и бьются на трети (приж / сбал / мощн) —
детерминированно, ровно по спреду и тематично (мощностная машина → мощностной концепт).
Концепт накладывается на машину ИИ в `main.buildField` через общий `applyConceptBias`
(симметрично, как у игрока), поэтому сильные стороны грида **меняются от трассы к трассе**.
Концепты видны во вкладке «Зачёт» (бейдж приж/сбал/мощн у каждой команды).

## E8 — экономика разработки ИИ
Блок разработки ИИ в `tickDevelopment` переписан:
- **ATR-догонялка по живому месту в чемпионате** — отстающие команды развиваются быстрее,
  лидеры медленнее (`atr = 0.55…1.05` по позиции из `career.teamPts`), как аэро-ATR у игрока;
  держит грид плотным.
- **Разброс формы** — сид-ролл на команду за период даёт прорыв (×1.45) / провал (×0.5) /
  норму; среднее нейтрально, поэтому общий темп прогресса сохранён, но появляются «взлёты и
  спады» сезонов у соперников.
- **Разработка под концепт** — мощностные команды качают `pu`+`sidepods` (прямые),
  прижимные `floor`+`fw` (повороты), сбалансированные `floor`+`pu`; их сильные стороны
  растут по их идентичности.
Средний прогресс — на паритете со старой плоской моделью (баланс-коридоры держатся).

## E9 — ресурс ДВС у ИИ
У каждой ИИ-команды свой пул ДВС (`career.aiPu[team]`, `PU_POOL=4`). За гонку их ДВС
изнашивается (`puWearForRace`, надёжность по силе завода: слабые команды быстрее);
перерасход → штраф мест на старте, который применяется к машинам этой команды по гриду
в `main.startRace` и попадает в ленту новостей («Ferrari: исчерпан лимит ДВС…»). Свежий
пул каждый сезон. Игрок видит, как соперники теряют позиции из-за моторных штрафов.

## F1 — фокус «этот сезон ↔ следующий» (MM-надстройка)
`career.devFocus` (0 / 0.25 / 0.5) — доля отдачи каждого готового апгрейда, которая
вместо текущей машины **банкуется в машину следующего года** (`career.nextCar`).
В `newSeason`, после сброса регламента и зимних тестов, задел добавляется к стартовым
деталям нового сезона (фора, которую регламент не срезает). Жертвуешь результатом сейчас
ради форы потом — самый «MM-ный» стратегический выбор. Слайдер в вкладке «Машина» +
строка «📦 Задел на след. год».

## F2 — пайплайн проект → сборка → установка (MM-надстройка)
Каждый проект теперь проходит две стадии: **проектирование** (`days`, R&D) → когда
дизайн готов, исход (прирост+надёжность) **прокатывается и фиксируется** (видно, что
получил) → **сборка** (`buildDays`, производство) → **установка**: прирост применяется
(с расщеплением F1) и деталь получает долг обкатки. ETA показывает стадию:
«📐 проект N дн → к ГП X» или «🔧 сборка M дн → … · +X.X (прорыв)». Стыкуется с обкаткой
(E1): свежеустановленная деталь — необкатанная.

Заметка: дискретный инвентарь деталей из MM сознательно НЕ вводился — у нас «уровень +
потолок» (E1), что покрывает эффект без риска разбалансировать сим и сейвы. Схема v21.
